Chapter One

Why this app exists

Online reviews are the modern storefront window. People decide whether to walk in based on what they see — and they notice whether you respond.

Studies of consumer behavior consistently show two things: most customers read reviews before choosing a business, and a visible, timely reply from the owner increases trust even when the original review is critical. The hard part isn't caring — it's keeping up. Reviews arrive at unpredictable hours, the good ones deserve acknowledgement, and the bad ones need a fast, human response before they fester.

The Google Reviews Dashboard exists to close that gap. It is a small, self-hosted full-stack application that does three jobs well:

  • Watches your Google reviews and pulls them into one dashboard.
  • Replies automatically to positive reviews using templates you control.
  • Alerts you the moment a low-rating review needs a human touch.
Design principle Automate the routine, escalate the exceptions. Five-star reviews get a warm template reply in seconds; one- to three-star reviews are flagged and emailed to a real person.

Who it's for

Owners of single-location or small multi-location businesses, agencies managing reputation for clients, and developers who want a transparent, no-black-box reply system they can host themselves. There is no SaaS subscription and no third-party data sharing — you run it, you own the data.

What makes it different

No LLM dependency Template-based replies Self-hosted SQLite — zero DB ops Open API

Auto-replies use deterministic templates, not a language model. That means no per-message cost, no surprising AI output, and replies you can predict and approve in advance. Everything runs in a single container if you want it to, backed by a file-based SQLite database.

Chapter Two

How it works at a glance

Two services, one database, one scheduled job. That's the whole machine.

The two halves

The backend is a FastAPI application with a SQLite database and an APScheduler-driven background job. The frontend is a React dashboard styled with Tailwind. In production they can be packaged into a single container, or run separately with Docker Compose.

├── backend/          # FastAPI + SQLite + APScheduler
│   └── app/
│       ├── routers/    # reviews, config, oauth endpoints
│       ├── services/   # Google Places, GBP API, reply engine
│       └── scheduler/  # poll + auto-reply job
└── frontend/         # React + Tailwind dashboard
    └── src/pages/    # Dashboard, Reviews, Settings

The data flow

Every sync follows the same path, whether it's triggered by the scheduler or by hand:

  1. Fetch. If Google Business Profile is connected, reviews come from there; otherwise the app falls back to the Places API.
  2. Upsert. Each review is matched by its external ID and inserted or updated. New reviews are counted.
  3. Reconcile existing replies. If Google already shows a reply, the local record is marked posted so the app never double-replies.
  4. Alert. Reviews rated 3 stars or below with no reply trigger a low-rating email alert.
  5. Auto-reply. Reviews at or above your threshold get a template reply, posted live if Google is connected or saved as pending if not.
The golden rule A review is only ever acted on when its reply_status is none. Once it's posted, pending, or failed, the sync skips it. This is what keeps the job idempotent — you can run it as often as you like.

Chapter Three

Getting it running

Three ways to start: bare-metal dev, a single container, or Compose. Pick the one that matches your moment.

Local development

Run the backend and frontend in two terminals. This gives you hot reload on both sides.

Backend

cd backend
pip install -r requirements.txt
uvicorn app.main:app --host 127.0.0.1 --port 8011 --reload

Frontend

cd frontend
npm install
npm run dev

Then open http://127.0.0.1:5173. The Vite dev server proxies API calls to the backend on port 8011.

✦ ✦ ✦

Single container (Docker)

For a self-contained deployment, build the image and run it with a named volume so your database survives restarts.

docker build -t google-reviews .
docker run -d \
  --name google-reviews \
  -p 127.0.0.1:8080:80 \
  -v google-reviews-data:/app/data \
  google-reviews

Open http://127.0.0.1:8080. The container serves the built React app and proxies the API behind it.

Don't skip the volume Without -v google-reviews-data:/app/data, your reviews, replies, and settings live only inside the container and vanish when it's removed. The volume is your database.

Docker Compose (separate services)

Compose runs the frontend and backend as distinct containers — closer to a scaled production layout.

cp .env.example .env   # fill in your keys
docker compose up --build

Copy the example environment file first and fill in any keys you want baked in. You can also enter everything later through the Settings page.

Chapter Four

Connecting Google

There are two integrations, and they do different jobs. Reading reviews and replying to them are governed by separate Google APIs.

Part one — Places API (reading)

The Places API is the quickest way to start seeing reviews. It requires only an API key and a Place ID.

  1. Enable Places API in the Google Cloud Console.
  2. Create an API key and paste it into Settings.
  3. Find your Place ID with Google's Place ID Finder.
A real limitation The Places API returns only the 5 most relevant reviews — not your full history, and you cannot post replies through it. For complete access and live replying, connect Google Business Profile below.

Part two — Business Profile (replying)

To publish replies that actually appear on Google, connect the Google Business Profile (GBP) API via OAuth.

  1. Enable the Google Business Profile API in Cloud Console.
  2. Create OAuth 2.0 credentials of type Web application.
  3. Add the redirect URI: http://127.0.0.1:8011/api/oauth/callback
  4. In Settings, enter your Client ID, Client Secret, Account ID, and Location ID.
  5. Click Connect Google and complete the consent screen.
Graceful degradation If GBP is not connected, the app still works. Replies are generated and saved locally with the status Pending publish — ready to go live the moment you connect.

How the app chooses a source

At sync time the logic is simple and predictable: if GBP is configured, the app reads reviews from there; if that returns nothing (or GBP isn't set up), it falls back to the Places API. You get the richest source available without having to choose manually.

Chapter Five

The auto-reply machine

This is the heart of the app: a deterministic engine that turns a star rating into the right reply, and a scheduler that keeps it running while you sleep.

Templates by rating

The engine picks a template based on the review's star count. Each is fully customizable in Settings, and three placeholders are filled in automatically.

RatingDefault template
5 ★Thank you so much, {author_name}! We're thrilled you had a great experience at {business_name}.
4 ★Thanks for the kind words, {author_name}! We appreciate your feedback at {business_name}.
3 ★ & belowThank you for your feedback, {author_name}. We'd love to hear how we can improve at {business_name}.

The available placeholders are {author_name}, {rating}, and {business_name}. Anything else in your template is passed through untouched.

The threshold

Auto-reply only fires when two conditions are met: the feature is enabled, and the review's rating is at or above your minimum (default 4). Anything below the threshold is deliberately not auto-replied — it's left for a human, and flagged.

def should_auto_reply(rating: int) -> bool:
    if auto_reply_enabled != "true":
        return False
    return rating >= auto_reply_min_rating   # default 4

Posting, or not

Once a reply is generated, the engine attempts to post it to Google. The outcome decides the stored status:

SituationStored status
Posted to Google successfullyposted
GBP configured but post failedfailed
GBP not connectedpending (saved locally)

The scheduler

An APScheduler job runs the whole sync-and-reply cycle on an interval you set (default every 4 hours, configurable from 1 to 168). You can also trigger it by hand at any time from the dashboard or via the API.

Why polling, not webhooks Polling keeps the system simple and self-contained — no public callback URL, no webhook registration, no inbound security surface. For review volumes typical of small businesses, a few-hour cadence is more than responsive enough.

Chapter Six

Low-rating alerts

The reviews that matter most are the ones the machine deliberately won't answer. Those get a human, fast.

When a review arrives rated 3 stars or below and has no reply yet, the app sends an email alert so a real person can craft a thoughtful, personal response. This is the deliberate counterweight to automation: warmth where it's needed, speed where it isn't.

What the alert contains

  • The star rating, rendered as ★ symbols
  • The reviewer's name
  • The full review text
  • A prompt to log in and reply manually

Setting it up

Alerts are optional and require SMTP credentials. In Settings, provide:

FieldPurpose
alert_emailWhere alerts are sent
smtp_hostMail server (default smtp.gmail.com)
smtp_userSMTP username / from address
smtp_passSMTP password or app password
Fails quietly by design If any SMTP field is missing, the alert is silently skipped — it never blocks a sync. And if sending throws an error, the job logs a warning and carries on. Reviews are still recorded; you just won't get that email.
Gmail tip For Gmail, generate an app password rather than using your account password, and keep two-factor authentication on. The default port is 587 with STARTTLS.

Chapter Seven

The API reference

Everything the dashboard does, it does through this API — so anything the dashboard can do, your own scripts can do too.

MethodEndpointDescription
GET/api/reviews/List reviews
GET/api/reviews/statsDashboard stats
POST/api/reviews/syncFetch reviews + run auto-reply
POST/api/reviews/{id}/replyPost a manual reply
POST/api/reviews/{id}/auto-replyGenerate a template reply
POST/api/trigger-syncManually trigger the scheduler
GET/api/oauth/connectStart Google OAuth
GET / POST/api/config/Read or update settings

A typical sync response

Calling /api/reviews/sync returns a summary of exactly what the run did:

{
  "fetched": 5,
  "new_reviews": 2,
  "auto_replied": 1,
  "alerts_sent": 1,
  "pending_replies": 3
}

Each field tells a small story: how many reviews were pulled, how many were genuinely new, how many got an automatic reply, how many low-rating alerts went out, and how many reviews are still waiting on you.

Manual replies

A manual reply takes a single field, validated for length:

POST /api/reviews/42/reply
{ "reply_text": "Thank you for visiting — see you again soon!" }

Reply text must be between 1 and 4096 characters. If GBP is connected, it posts live; otherwise it's saved as pending, exactly like an auto-reply.

Interactive docs Because the backend is FastAPI, a full interactive OpenAPI explorer is available at /docs on the backend port. Try every endpoint from the browser without writing a line of code.

Chapter Eight

Operating in production

The app is simple, but reputation data and Google credentials are sensitive. A few habits keep it safe and reliable.

Security

  • No built-in auth. The app ships without authentication. Never expose it directly to the open internet — put it behind a reverse proxy with auth, a VPN, or bind it to localhost as the examples do (127.0.0.1).
  • Protect the credentials. Your Places key, OAuth secret, and SMTP password live in the database and settings. Restrict who can reach the Settings page.
  • Scope your API key. In Cloud Console, restrict the Places API key to the Places API and, where possible, to your server's IP.

Backups

Your data is one file Everything — reviews, replies, settings, OAuth tokens — lives in the SQLite database under /app/data. Back up that volume regularly. Losing it means losing your reply history and reconnecting Google from scratch.

Good operating habits

  • Approve your templates first. Because replies are deterministic, read them aloud before enabling auto-reply. Whatever you write is exactly what customers will see.
  • Mind the threshold. Keeping the minimum at 4 means borderline 3-star reviews always reach a human — usually the right call.
  • Watch the alert inbox. The system is only as good as your response to the alerts it sends. Treat a low-rating email as a same-day task.
  • Tune the poll interval. Higher review volume warrants a shorter interval; a quiet location can poll less often to conserve API quota.
